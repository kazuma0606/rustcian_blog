# Phase 2.0 Tasks: Content Operations and Rich Rendering

## 1. ゴール
- 公開ブログを読み取り専用の安全な構成として維持できること。
- Git ベースで下書きと公開を管理できること。
- 記事本文と運用メタデータを分離し、記事ディレクトリ単位で管理できること。
- Markdown 表示品質を改善し、LaTeX 数式を安定表示できること。
- CSV などのデータアセットから図表を生成できること。
- 将来の管理機能と AI 補助機能を Azure 前提で拡張しやすい境界が整理されていること。

## 2. スコープ
- 対象: 記事ディレクトリ構成、`post.md` / `meta.yml`、公開判定、表示品質改善
- 対象: KaTeX ベースの数式表示、コードブロック改善、画像やメタ情報の表現強化
- 対象: 記事アセットとしての CSV と図表仕様、図表コンポーネントの導入
- 対象: Azure OpenAI / Entra ID のためのアダプタ境界と PoC
- 非対象: Azure 本番デプロイ workflow の完成、一般公開向け投稿 UI の実装

## 4. 完了条件
1. 記事を `content/posts/<slug>/` 単位で管理できる。
2. `meta.yml` の `status: draft | published` により公開対象を制御できる。
3. 公開面では `published` のみが一覧・詳細に表示される。
4. 数式入り Markdown を KaTeX ベースで安定表示できる。
5. 記事ディレクトリ配下の CSV を参照した図表を記事内に表示できる。
6. AI 補助メタを本文と分離した形で保存できる方針が固まっている。
7. 管理系機能を Entra ID 前提で拡張できる境界が整理されている。

## 5. 次フェーズへの引き継ぎ条件
- 公開面と管理面の責務分離が明確であること。
- `meta.yml` と補助メタデータの仕様が固まっていること。
- 数式表示と図表表示の基盤が安定していること。
- Azure OpenAI と Entra ID のアダプタ境界が定義されていること。
- 今後 Azure 本番実装を追加しても `application/core` の変更を最小化できること。
